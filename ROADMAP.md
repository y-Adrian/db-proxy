# DB-Proxy 演进路线图

本文档将产品能力按**四阶段**拆解为可跟踪的里程碑；粒度介于「口号」与「单 PR 任务」之间，便于排期与验收。当前主程序行为以 [README.md](README.md) 为准：**线协议透明中继 + 连接池槽位与预热**；与「SQL 级智能代理」相关的项在文中单独标注为**可选产品形态**，避免与现有透传语义混淆。

---

|------|------|------------------------|

## 第一阶段：基础设施与资源模型

**目标：** 在现有架构上把「监听—接入—池—会话」链路做成**可配置、可预期、易排障**的生产基线。

| 里程碑 | 交付物 / 范围 | 验收要点（建议） |
|--------|----------------|------------------|
| **1.1 监听与资源上限** | `[server] max_connections`、`backlog` 等配置**实际接入** `EpollServer::listen` / accept 路径 | 压测或配置极限下行为符合配置；有日志说明拒绝原因 |
| **1.2 连接池行为文档化** | 明确「槽位 / 会话 / 后端连接」生命周期；与 README 一致或修正文档 | 新人可按文档理解借还顺序与失败回滚 |
| **1.3 健康检查与失效策略** | 池内 ping/重连策略与指标对齐（失败率、剔除间隔） | 拔后端网线或关库时，池状态可观测且可恢复 |
| **1.4 RAII 与错误路径审计** | socket、池连接、线程退出路径统一；关键路径无泄漏/双关 | ASan/LSan 或长期跑压无资源泄漏报告 |
| **1.5 非 Linux 行为** | `select` 回退路径的 FD 上限与文档说明 | macOS/其他平台编译与基础功能测试清单 |

**依赖：** 无（当前阶段主体）。  
**说明：** 本阶段**不改变**「透明中继」语义，主要为工程化与正确性。

---

## 第二阶段：协议与可观测性

**目标：** 在**不强制改变**主路径的前提下，增强「能看见什么、能基于什么做决策」；为第三阶段路由与第四阶段性能优化提供数据面。

| 里程碑 | 交付物 / 范围 | 验收要点（建议） |
|--------|----------------|------------------|
| **2.1 指标与统计统一** | `Metrics` / `Statistics` / 日志字段对齐同一套语义（QPS、会话时长、池占用） | Grafana/Prom 或本地 `curl /metrics` 可解释；与诊断模块输入一致 |
| **2.2 慢路径与追踪关联** | 会话级慢阈值已有；可选增加 trace id / 连接 id 贯穿日志 | 单次慢会话可从日志串到池状态快照 |
| **2.3 SQL 解析能力（库路径）** | 复用/增强 `mysql_parser` 等：从**示例或可选模块**输出「语句类型 + 指纹」（不落主路径亦可） | 单元测试覆盖常见语句类型；性能基准有数字 |
| **2.4 主路径策略开关（可选）** | 定义配置项：`relay_only`（默认） vs `sql_aware`（实验性） | 默认行为与今日二进制一致；开启后行为与文档一致 |
| **2.5 预编译语句（MySQL COM_STMT）** | 仅在 **sql_aware** 或独立工具路径实现：解析、缓存键、失效策略 | 与官方客户端兼容矩阵列出；有回归测试集 |
| **2.6 诊断与运维联动** | `diagnostics_demo` / 未来 HTTP 触发与 metrics 同源 | 同一份统计快照可复现诊断报告 |

**依赖：** 第一阶段 1.2–1.3 建议完成后再大规模改统计语义。  
**风险：** **2.4–2.5** 与「透明中继」并存时，需严格版本与配置说明，避免用户误以为默认会解析 SQL。

## 第三阶段：扩展性、高可用与运维

**目标：** 多后端、可切换、少重启；与 `PoolManager` 多池模型对齐。

| 里程碑 | 交付物 / 范围 | 验收要点（建议） |
|--------|----------------|------------------|
| **3.1 多后端池配置模型** | INI 或等价配置支持多 `[database.*]` / 池别名；与现有 `primary` 兼容迁移 | 提供示例 conf；旧配置无需改即可运行 |
| **3.2 读写分离（路由层）** | 将 `examples` 中路由思路提升为**可配置策略**：只读池集合、写默认主池、可选粘性 | 只读流量进从池；写与事务进主池；文档写明不支持的场景（如驱动级事务语义） |
| **3.3 负载均衡** | 读池间轮询/随机/最少连接；健康池剔除与恢复 | 摘一台从库后读请求迁移；恢复后自动纳回 |
| **3.4 配置热更新** | 至少一种机制：SIGHUP、HTTP admin reload、或周期性重载文件 | 改池地址/权重不重启进程即可生效；失败时保留旧配置并打错误日志 |
| **3.5 运维接口（可选）** | 只读 admin：池状态、版本、最后 reload 时间 | 鉴权方式（本地回环 / token）在文档中固定 |

**依赖：** 第二阶段 2.1–2.2 强烈建议完成（否则路由与健康缺少观测）。**3.2** 若坚持主路径仍为纯透传，可仅交付「应用侧池 + 多端口代理」方案并在文档中界定边界。

---

## 第四阶段：高性能内核（可选 / 长期）

**目标：** 在**已证明瓶颈在 I/O 或调度**的前提下，替换或并行实现高性能路径；不作为默认承诺。

| 里程碑 | 交付物 / 范围 | 验收要点（建议） |
|--------|----------------|------------------|
| **4.1 性能基线与剖析** | 固定 workload、火焰图、epoll vs 阻塞中继瓶颈报告 | 数据驱动决策是否进入 4.2+ |
| **4.2 io_uring（Linux）** | 限定 Linux：接受/读写在实验分支或编译开关 | 同 workload 下延迟/吞吐对比；失败回退 epoll |
| **4.3 无锁或少锁结构** | 连接表、队列等热点经 profiling 后替换 | 正确性测试 + 压力测试；文档说明内存序假设 |
| **4.4 协程化会话（可选）** | 与线程池对比：栈、取消、与 OpenSSL/驱动交互边界 | 不劣化默认路径稳定性 |
| **4.5 NUMA 亲和** | 多插槽机器上绑核/内存策略 | 仅在部署文档与可选启动参数中启用 |

**依赖：** 第三阶段稳定运行；**禁止**在无基线情况下直接切换全局 I/O 模型。

## 跨阶段事项（全程）

| 项 | 说明 |
|----|------|
| **测试** | 每里程碑合并前：`ctest`、与里程碑相关的集成脚本；涉及协议变更时补充抓包或兼容矩阵 |
| **文档** | README 与 ROADMAP 同步「默认行为 vs 实验功能」；配置项变更提供迁移说明 |
| **安全** | admin、热更、多后端均默认最小暴露面（回环、鉴权） |
| **兼容性** | MySQL/PG 线协议版本与驱动组合在 README 或单独 compatibility 表中维护 |

## 建议排期顺序（滚动规划）

1. **先做** 一阶段 1.1、1.3 与 二阶段 2.1、2.2（性价比最高，风险低）。  
2. **再定** 三阶段 3.1、3.4（运维刚需，且与多池模型天然契合）。  
3. **然后** 3.2、3.3（与 2.3/2.4 是否启用 SQL 感知强相关）。  
4. **2.5、第四阶段** 单独立项，避免与主路径稳定性抢同一迭代。
